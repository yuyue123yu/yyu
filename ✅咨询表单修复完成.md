## 🐛 问题描述

### 错误信息
```
null value in column "tenant_id" of relation "consultations" violates not-null constraint
```

### 问题原因
咨询表单提交时没有设置 `tenant_id` 字段，导致数据库约束错误。

### 影响范围
- ❌ 用户无法提交咨询请求
- ❌ 首页咨询功能不可用
- ✅ 其他页面功能正常

---

## 🔧 修复方案

### 修复内容
在 `src/app/consultation/page.tsx` 中添加了获取默认租户的逻辑：

```typescript
// 修复前
const { error: insertError } = await supabase
  .from('consultations')
  .insert([
    {
      client_id: user?.id || null,
      name: formData.name,
      // ... 其他字段
      // ❌ 缺少 tenant_id
    },
  ])

// 修复后
// Get the first active tenant (temporary solution)
const { data: tenant } = await supabase
  .from('tenants')
  .select('id')
  .eq('status', 'active')
  .order('created_at', { ascending: true })
  .limit(1)
  .single()

if (!tenant) {
  throw new Error('No active tenant found')
}

const { error: insertError } = await supabase
  .from('consultations')
  .insert([
    {
      tenant_id: tenant.id, // ✅ 添加了 tenant_id
      client_id: user?.id || null,
      name: formData.name,
      // ... 其他字段
    },
  ])
```

### 修复逻辑
1. 在提交咨询前，先查询第一个活跃的租户
2. 获取该租户的 ID
3. 将租户 ID 设置到咨询记录中
4. 如果没有找到活跃租户，抛出错误

## 📝 技术说明

### 临时解决方案
这是一个**临时解决方案**，适用于当前单租户场景。

**优点**:
- ✅ 快速修复问题
- ✅ 不需要重新启用 middleware
- ✅ 适合当前单租户使用

**限制**:
- ⚠️ 如果有多个租户，无法自动区分
- ⚠️ 总是使用第一个活跃租户

### 长期解决方案
未来需要重新启用 middleware 来实现真正的多租户支持：

1. **简化 middleware 逻辑**
   - 只保留基本的租户识别
   - 避免复杂的数据库查询

2. **通过域名识别租户**
   - 自定义域名 → 对应租户
   - 子域名 → 对应租户
   - 默认域名 → 第一个租户

3. **在请求头中传递租户 ID**
   - middleware 设置 `x-tenant-id` header
   - API 和页面从 header 读取租户 ID
